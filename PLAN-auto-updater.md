# Plan: Tauri Auto-Updater for ProveAll

## Overview

Add auto-update functionality using `tauri-plugin-updater` so users are notified when a new version is available, with an optional setting to auto-install updates.

## Current State

- **Version**: 0.4.1, tracked in 4 places (package.json, tauri.conf.json, Cargo.toml, CHANGELOG.md)
- **Releases**: Automated via GitHub Actions on push to `main`. Builds macOS aarch64 `.dmg` and `.app.tar.gz`, creates a GitHub Release with version tag.
- **Settings**: Existing `SettingsPage.jsx` with LLM config. Settings persisted to `~/.local/share/com.proveall/settings.json` via Rust backend.
- **Frontend**: React 18 with state-based routing (no React Router). Tauri commands called via `window.backendAPI` wrapper in `tauriAPI.js`.

## What the Updater Plugin Needs

The `tauri-plugin-updater` checks a remote endpoint for update info. It needs a JSON response in this format:

```json
{
  "version": "0.5.0",
  "notes": "Release notes here",
  "pub_date": "2024-01-01T00:00:00Z",
  "platforms": {
    "darwin-aarch64": {
      "signature": "...",
      "url": "https://github.com/.../ProveAll_0.5.0_aarch64.app.tar.gz"
    }
  }
}
```

The app must be code-signed, and the `.app.tar.gz` must have a corresponding `.sig` signature file (generated by Tauri's build process when a signing key is configured).

## Required Changes (Regardless of Decisions Below)

### 1. Add Tauri Updater Plugin

- Add `tauri-plugin-updater` to `tauri_src/Cargo.toml`
- Register the plugin in `tauri_src/src/lib.rs`
- Add `@tauri-apps/plugin-updater` to `package.json`
- Configure updater endpoint in `tauri.conf.json`

### 2. Generate Update Signatures

- Generate a Tauri updater keypair (`tauri signer generate`)
- Store the private key as a GitHub secret (`TAURI_SIGNING_PRIVATE_KEY`)
- Set the public key in `tauri.conf.json` under `plugins.updater.pubkey`
- The CI build will then produce `.sig` files alongside the `.app.tar.gz`

### 3. Add "Auto Update" Setting

- Extend the settings JSON to include `auto_update: bool` (default: `false`)
- Add a toggle to `SettingsPage.jsx` under a new "Updates" section
- Persist via the existing settings mechanism (Rust backend read/write)

### 4. Update Notification UI

- On app launch, check for updates
- If update available and `auto_update` is off: show a non-blocking notification/banner (e.g., top bar or toast) with "Update available — v{version}. [Install Now] [Dismiss]"
- If `auto_update` is on: download and prompt to restart

### 5. CI Changes

- Modify `release.yml` to set `TAURI_SIGNING_PRIVATE_KEY` env var during build so `.sig` files are generated
- Upload `.sig` files as release assets alongside existing artifacts
- Generate the updater JSON manifest (see Decision 1 below)

## Decision 1: Update Endpoint Strategy

The updater plugin needs a URL that returns the JSON manifest above. There are two approaches:

### Option A: Static JSON File on GitHub Releases

**How it works**: The CI workflow generates a `latest.json` file during the release process and uploads it as a release asset. The updater endpoint points to a stable URL like:
```
https://github.com/user/proveall/releases/latest/download/latest.json
```

**Pros**:
- Zero infrastructure — no server to maintain
- Free, uses existing GitHub Releases
- Simple CI change: just generate and upload one extra file
- Perfect for open-source projects

**Cons**:
- GitHub CDN may cache the file (users might see stale versions briefly)
- No analytics on update checks
- Less flexible (can't do staged rollouts, A/B, etc.)

### Option B: Dynamic Update Server (e.g., a Cloudflare Worker, Vercel function, or self-hosted)

**How it works**: A small server/function reads from GitHub Releases API and returns the updater JSON. The updater endpoint points to your server URL.

**Pros**:
- No caching issues — always fresh
- Can add analytics, rate limiting, staged rollouts
- Can add platform-specific logic

**Cons**:
- Extra infrastructure to maintain and pay for
- More complex setup
- Overkill for a public open-source project with straightforward releases

**Recommendation context**: Most open-source Tauri apps use Option A. Tauri's own documentation shows the static file approach for GitHub-based projects. Option B is typically used by commercial apps that need rollout control.

> **Action required**: Which approach do you want? **A (static file on GitHub Releases)** or **B (dynamic update server)**?

## Decision 2: Update Logic Location — JavaScript vs Rust

The updater plugin exposes APIs on both the JS and Rust sides. The question is where the "check for update → download → install" logic lives.

### Option A: JavaScript Side (Frontend)

**How it works**: Use `@tauri-apps/plugin-updater` JS API in React. The frontend handles checking, showing UI, and triggering install.

```js
import { check } from '@tauri-apps/plugin-updater';

const update = await check();
if (update?.available) {
  // Show notification in React UI
  // User clicks "Install" → await update.downloadAndInstall();
}
```

**Pros**:
- Update check + UI logic live together — simpler flow
- Full control over UX (progress bars, toasts, etc.) without IPC
- Less Rust code to write
- Matches the project's existing pattern (frontend drives actions, calls backend for data)

**Cons**:
- Update logic is coupled to the UI layer
- If you later want background/headless update checks, you'd need to move logic

### Option B: Rust Side (Backend)

**How it works**: Rust code checks for updates on app startup, emits Tauri events to the frontend to display notifications. Frontend sends commands back to trigger download/install.

```rust
// In Rust setup:
let update = app.updater().check().await?;
if update.is_some() {
    app.emit("update-available", payload)?;
}

// Frontend listens:
listen("update-available", (event) => { /* show UI */ });
// Frontend triggers: invoke("install_update");
```

**Pros**:
- Update logic is independent of UI lifecycle
- Could check in background before window is fully loaded
- Cleaner separation of concerns

**Cons**:
- More boilerplate (Rust commands + event emitters + JS event listeners)
- The existing codebase pattern is "frontend calls backend commands" — events are a different pattern
- More Rust code for what is essentially UI-driven behavior

**Recommendation context**: The existing project pattern is frontend-driven — `tauriAPI.js` calls `invoke()` for everything. The JS approach fits this pattern naturally. The Rust approach is better if you anticipate needing background update checks or headless operation.

> **Action required**: Which approach do you want? **A (JavaScript/frontend-driven)** or **B (Rust/backend-driven)**?

## Implementation Steps (After Decisions)

Once the two decisions above are made, implementation follows this order:

1. **Generate signing keypair** and add private key to GitHub secrets
2. **Install plugin dependencies** (Cargo + npm)
3. **Configure `tauri.conf.json`** with updater endpoint and public key
4. **Extend settings model** to include `auto_update` boolean
5. **Add update check logic** (in JS or Rust depending on Decision 2)
6. **Add UI components** — notification banner + settings toggle
7. **Update CI workflow** — signing key env var + manifest generation (depending on Decision 1)
8. **Test end-to-end** with a real release
